## 2. Required implementation order

1. D1–D4 establish safe identity, navigation and durable work before adding new mutations.
2. D5 establishes a real MySQL transaction boundary before D6 may reach a live server.
3. D6 and D9 complete the mutation and inspection loops (D7/D8 deferred to P1 by v1.1).
4. D10 and D11 are tested continuously and gate every slice, not deferred as packaging or decoration.
5. D12 starts only after D1–D11 are green and independently reviewed.

## 3. Cross-row invariants

- Generation, session and operation identity must be checked at every asynchronous completion.
- No Rust mutex/RwLock guard or borrowed in-process lock guard crosses `await`; stateful database ownership uses an async worker/task. DU-05's transaction-lifetime and DU-07's operation-lifetime nonblocking OS advisory leases are owned by the synchronous safety-lock component and are not in-process data guards.
- No credential or non-fixture user query/key/cell/CSV content appears in `Debug`, tracing, public errors or release evidence. DU-11 public screenshots may contain only the exact tracked synthetic visual-fixture allowlist from an isolated temporary config; every other value and the forbidden sentinel fail capture.
- All user-controlled SQL identifiers are validated and quoted; all values use typed parameters. Daily-use v1 rejects implicit-commit DDL/admin and raw transaction/session controls before user-target I/O, after only the explicitly allowed static capability metadata reads.
- Every mutation is rejected for Read-only before user-target/typed-mutation dispatch; the bounded static server/session/catalog capability reads explicitly allowed by DU-01/DU-03 may acquire a metadata-only lease.
- Production destructive actions require a review and exact confirmation where specified.
- All dbotter-retained lists/files are bounded before retention and associated server writes; driver frame/row transient allocation remains disclosed and is still cut off at retained result limits.
- File writes are atomic and fail closed on uncertain durability.
- A stale, cancelled or mismatched event cannot replace the active tab, result, transaction, stage or review state.
